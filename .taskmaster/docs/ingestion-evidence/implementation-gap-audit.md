# Implementation Gap Audit — Source Requirements vs Current Code

Author: **Manus AI**

This audit was produced only after the source-first ingestion was completed. It maps the combined requirement register to the current implementation evidence read from `Home.tsx`, `TerminalPanel.tsx`, `FilesystemPanel.tsx`, `wrapperLLM.ts`, `routers.ts`, and `drizzle/schema.ts`.

## Audit Summary

The current application has a meaningful foundation for the WEBpORTAL concept: authenticated tasks, task events, global memory, task files, credential snapshots, tag-based route overrides, and direct Claude/Kimi calls are present. However, the implementation still fails the most important product invariant from the uploaded documents: **the owner should not see or operate a terminal by default**. The current UI mounts an authenticated xterm terminal directly in the right sidebar, and the terminal opens a WebSocket connection on render.

The second major gap is that the current backend is an **LLM response orchestrator**, not yet a hidden Kimi/OpenCode execution orchestrator. Kimi is invoked through Cloudflare Workers AI and instructed to describe patch strategy rather than actually apply repository changes. The source documents require Kimi to operate as the execution/build side behind the scenes, with Claude planning and reviewing and the owner seeing plain-English updates.

## Confirmed Fix Targets

The immediate implementation fixes should be narrow and evidence-backed. First, the default right sidebar must stop mounting the terminal. The owner should see a task file/context panel by default, with a clearly separated advanced control that mounts the terminal only after explicit action. Second, the owner-facing copy should remove confusing “Wrapper”/terminal-centric labels where they appear in primary UI controls. Third, the right sidebar should be simplified to reduce clipping: task-scoped files and master files should be visible first, while advanced filesystem editing and terminal access should be collapsed.

## Deferred or Scope-Decision Targets

The source documents require a deeper orchestration loop than the current app implements. Adding owner approval states, a true plan/build/review/decision lifecycle, and hidden Kimi/OpenCode execution is larger than a visual bug fix. These items should not be silently approximated. They require a product decision: either implement them now as part of the MVP, or explicitly mark current behavior as “planning and execution-draft orchestration” rather than real hidden build execution.
